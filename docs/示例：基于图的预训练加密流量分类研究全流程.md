## 阶段一：选题定向与研究框架搭建

### 场景设定

研究者注意到：随着 TLS 1.3 和 ECH（Encrypted Client Hello）的普及，传统基于明文字段（SNI、证书）的流量分类手段逐渐失效。同时，图神经网络在流量交互关系建模上展现出潜力，但标注数据稀缺是瓶颈。初步想法：**能否用图自监督预训练降低对标注数据的依赖？**

### Step 1: 调用 `/ars-plan` 凝练研究问题

**实际输入的指令**：

> /academic-research-suite: 我想研究「基于图神经网络的预训练加密流量分类」
>
> 背景：
> - TLS 1.3/ECH 使得 SNI 等明文特征不可用，传统 DPI 失效
> - 加密流量分类是入侵检测、QoS 调度、应用识别的上游基础任务
> - 图神经网络（GNN）可以建模流间交互（如 client-server 通信图），但现有工作多为监督学习，标注成本高
> - 图自监督预训练（如 GraphCL、GRACE）在分子图、社交网络等领域效果显著，尚未在流量图上充分探索
>
> 请梳理：核心研究问题、与现有工作的关键差异、初步创新方向

**AI 输出示例（节选）**：

```
核心研究问题：
  RQ1: 如何将加密流量会话构造为图结构，使其保留流间交互语义？
  RQ2: 针对流量图，哪些图自监督预训练任务（对比学习/掩码重建/边预测）能学到有效的流量表示？
  RQ3: 预训练得到的流量图表征能否跨数据集、跨应用类型迁移？

与现有工作差异：
  - 现有加密流量分类：侧重大规模监督学习（如 DeepPacket, FlowPrint），依赖标签 → 我们用自监督降低标签依赖
  - 现有图流量建模：仅用 GNN 做有监督分类 → 我们引入预训练-微调范式
  - 现有图预训练工作：聚焦分子图/社交网络 → 流量图有独特的时间衰减、burst 特性，需要定制预训练任务

初步创新方向：
  1. 提出流量交互图（Traffic Interaction Graph, TIG）构造方法
  2. 设计流量感知的图对比学习预训练策略（TIGraphCL）
  3. 建立跨数据集迁移评估协议
```

**实际输入的指令**：

> 映射到 MITRE ATLAS v5.4 框架，重点标注：
> - 对抗流量混淆（adversarial traffic obfuscation）：攻击者通过填充、分片、时序扰动改变流量特征

> - 概念漂移（concept drift）：模型训练后新应用/新协议出现，分类性能退化
>
> 分析每种威胁对「图预训练模型」的具体影响路径

**AI 输出片段**：

```
威胁 1：对抗流量混淆 vs. 图预训练模型
  - 攻击面：节点特征层（包大小序列被填充改变）、边结构层（分片改变流数 → 图拓扑变化）
  - 影响：若预训练仅依赖结构增强，攻击者可通过注入虚假边污染预训练表示
  - 缓解方向：在预训练阶段引入对抗增强（adversarial augmentation）

威胁 2：模拟攻击 vs. 图预训练模型
  - 攻击面：应用指纹层——恶意流量模仿目标应用的流量模式
  - 影响：图对比学习依赖应用级相似性区分，模拟攻击可降低类间差异
  - 缓解方向：融合有效载荷无关的时序微观特征（burst 模式、IAT 分布）

威胁 3：概念漂移 vs. 图预训练模型
  - 攻击面：时间维度的数据分布变化
  - 影响：预训练数据覆盖时期有限，新应用类型不在预训练表示空间中
  - 缓解方向：设计开放集检测 + 持续预训练机制
```

> 5. 代码实现的分阶段计划（PyTorch + PyG/DGL）

**AI 输出示例（方法架构片段）**：

```
TIGraphCL 整体架构（三阶段）：

Phase 1: 流量交互图构造
  输入: 原始 pcap
  处理: Zeek 解析 → 五元组流提取 → 流特征计算（包大小序列、IAT 序列、burst 统计、TLS 握手特征）
  图构造: 共享源 IP 的流 → 边相连，边权重 = 时间窗口内的流对相关性
  输出: Traffic Interaction Graph (TIG), G = (V, E, X, A)
    V: 流量会话节点，X_v ∈ R^{d_node}（流级统计特征）
    E: 共享源 IP 或相近时间窗口内的流对，A_{uv} = 时间衰减权重

Phase 2: 流量感知图对比预训练
  输入: 无标签 TIG 集合
  增强策略:
    - 流量属性增强：节点特征随机 mask（模拟丢包/重传）、高斯噪声注入（模拟抖动）
    - 图结构增强：边随机丢弃（模拟连接失败）、子图采样（模拟部分流量捕获）
    - 时间感知增强：时间窗口平移（模拟不同采集时段）
  对比目标: InfoNCE loss，正样本 = 同一原始图的两个增强视图，负样本 = 不同图的增强视图
  编码器: 3 层 GIN + 图池化 → 图级表示 h_G ∈ R^{256}

Phase 3: 下游微调
  输入: 预训练 GNN 编码器 + 少量标注 TIG
  微调: 冻结前 2 层 GIN → 仅微调第 3 层 + 分类头
  分类: 全连接层 → softmax → 应用类型/流量类型
```

### Step 2: 领域适配性校验

**实际输入的指令**：

> 从网络流量检测的专业角度，评估这套 TIGraphCL 方案的可行性
>
> 重点检查：
> 1. 流量图构造：五元组流提取在 NAT/CGN 环境下的准确性；图规模（单 host 可能产生数千条流）对 GNN 计算开销的影响
> 2. 预训练增强策略：节点特征 mask/噪声注入是否可能破坏 TLS 握手的语义完整性
> 3. 部署可行性：GNN 推理延迟是否满足实时 IDS 场景（<10ms per flow）的约束
> 4. 对抗鲁棒性：攻击者是否可以通过精心构造流量（注入虚假流、延迟真实流）来破坏图结构

**AI 输出片段**：

```
可行性评估：

✅ 可行的点：
  - 五元组流提取成熟（Zeek/NFStream 标准化输出），工程风险低
  - 图对比学习框架在分子图、社交网络的代码基础可复用（PyG 内置 GraphCL）

❌ 不可行的点：
  - 无（方案核心逻辑成立，风险可控）
```

### 子阶段 4.1：代码实现与小规模原型验证

#### Step 1: 代码架构设计

**实际输入的指令**：

> 参考 Zeek 日志分析最佳实践，设计 TIGraphCL 的代码架构
>
> 模块划分：

## 全流程小结

### 各阶段 AI 参与度与人工主导度

| 阶段 | AI 角色 | 人工必须主导 |
|------|---------|-------------|
| 1. 选题定向 | 梳理问题边界、提出结构化框架 | 判断研究方向价值、敲定研究边界 |
| 2. 文献调研 | 检索、分类、生成缺口矩阵 | 筛选文献质量、确认缺口合理性 |
| 3. 方案设计 | 生成方案框架、方法细节 | 敲定技术路线、判断方案创新性 |
| 4. 实验验证 | 代码生成、调参、图表绘制 | 判断实验结果合理性、定位问题 |
| 5. 模拟审稿 | 模拟审稿意见、安全分析 | 评估意见优先级、决定修改范围 |
| 6. 论文写作 | 逻辑校验、润色、格式检查 | 撰写核心论证、审核技术表述 |
| 7. 返修归档 | 回复框架、二次校验、归档 | 确定回复策略、确认归档完整性 |

### 关键经验

1. **不要跳过阶段五（写作前审稿）**：TIGraphCL 案例中，写作前审稿揭示了「创新性表述不够突出」和「缺少 ET-BERT 基线」两个关键问题，如果投稿后再发现将会是拒稿理由
2. **威胁建模（阶段二）带来意外收获**：从攻击者视角审视发现了「图结构投毒」攻击面，最终成为论文的加分实验
3. **代码审计（阶段四）避免致命错误**：`train_test_split(shuffle=True)` 导致的 data leakage 是投稿中常见的拒稿原因
4. **图表预规划（阶段三）节省返工**：在方案设计阶段确定了 7 图 2 表，后续实验和可视化直接按规划执行，避免了"数据不够画图"的尴尬
5. **条件 Skill 判断**：本项目是 GNN + 自监督学习的方法研究，明确跳过了 YARA 和 Sigma 相关 Skill，把精力集中在真正需要的工具上

### 实际发生的迭代

本案例并不像上文这样一帆风顺。以下是实际研究中发生的几次主要回退：

| 回退 | 触发原因 | 处理 |
|------|----------|------|
| 阶段四.2 → 阶段三 | 大规模实验中 GIN 在 VPN-Web 和 VPN-Streaming 两类上严重混叠 | 回到方案设计，增加流指纹相似度边过滤机制 |
| 阶段五 → 阶段四.2 | 模拟审稿指出缺少 ET-BERT 和 FlowTransformer 基线 | 补充 2 周实验，增加两组基线对比 |
| 阶段六 → 阶段四.2 | 写 Discussion 时发现无法解释跨数据集迁移掉至 0.734 的原因 | 补充 USTC-TFC2016 数据分布分析，确认原因后写入 Discussion |

> **结论**：三次回退总共增加了约 4 周工作量，但如果在投稿后再由审稿人指出这些问题，轻则 Major Revision（+3 个月），重则直接拒稿（+6 个月重投）。**在审稿前自己发现问题，永远是最划算的。**
